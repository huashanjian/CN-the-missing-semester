命令行环境

Command-Line Environment

作业控制 (Job Control)

在 Shell 中工作时，我们经常需要管理正在运行的多个进程。作业控制允许我们中断、暂停、并在后台或前台恢复这些进程。

终止进程 (Killing a process)

当你需要停止一个命令（例如它花费了太长时间），通常可以按 Ctrl-C。这会向进程发送一个 SIGINT (中断信号)。

Shell 使用 信号 (signal) 这种 UNIX 通信机制向进程传递信息。信号是软件中断，进程接收到信号后会暂停当前执行，处理信号，并可能改变其执行流程。

有时进程会捕获 SIGINT 并选择不停止。在这种情况下，你可以使用 Ctrl-\ (Ctrl + 反斜杠) 发送 SIGQUIT 信号，这通常会强制进程退出并可能生成一个核心转储文件用于调试。

#!/usr/bin/env python
import signal, time

def handler(signum, frame):
    print("\nI got a SIGINT, but I am not stopping")

signal.signal(signal.SIGINT, handler)
i = 0
while True:
    time.sleep(.1)
    print("\r{}".format(i), end="")
    i += 1

上述 Python 脚本会捕获 SIGINT。用 Ctrl-\ (SIGQUIT) 可以终止它。

常用信号总结:

  • SIGINT (2): 中断信号。由 Ctrl-C 发送。可被捕获，通常请求程序优雅停止。
  • SIGQUIT (3): 退出信号。由 Ctrl-\ 发送。通常导致进程终止并进行核心转储。
  • SIGTERM (15): 终止信号。 kill 命令默认发送此信号。标准请求进程终止，可被捕获以进行清理。(kill -TERM <PID> 或 kill -15 <PID>)
  • SIGKILL (9): 强制终止信号。不可被捕获、阻塞或忽略。立即终止进程，应作为最后手段 (kill -9 <PID>)。
  • SIGSTOP (19): 停止信号 (不可捕获)。暂停进程执行，类似 Ctrl-Z 但不可忽略。
  • SIGTSTP (20): 终端停止信号。由 Ctrl-Z 发送。可被捕获。
  • SIGCONT (18): 继续信号。使已停止的进程恢复执行 (bg, fg 命令发送)。
  • SIGHUP (1): 挂起信号。终端关闭时发送给关联进程。常用于守护进程重载配置。

更多信息请查阅 man 7 signal 或 kill -l。

暂停和后台运行进程 (Pausing and backgrounding processes)

除了终止，信号也可以暂停进程。例如，SIGSTOP 会暂停进程。在终端中，按 Ctrl-Z 会发送 SIGTSTP (终端停止信号)。

已暂停的作业可以使用 fg (foreground) 在前台恢复，或使用 bg (background) 在后台恢复。

jobs 命令列出当前终端会话中未完成的作业。你可以通过 PID (用 pgrep 查找) 或作业号 (%N，由 jobs 显示) 来引用它们。特殊参数 $! 引用最后一个置于后台的作业。

在命令末尾添加 & 后缀会在后台运行该命令，并立即返回提示符。但它仍会使用 Shell 的 STDOUT，建议使用重定向：

my_long_command > output.log 2> error.log &

要将已运行的程序置于后台，可以按 Ctrl-Z 然后输入 bg。注意，后台进程仍是终端的子进程，关闭终端会发送 SIGHUP 终止它们。

为防止关闭终端时后台进程被终止，可使用 nohup (忽略 SIGHUP 的包装器) 运行程序，或对已启动的进程使用 disown。终端多路复用器 (如 tmux) 是更高级的解决方案。

示例会话:

$ sleep 10000
^Z # 按 Ctrl-Z
  + 18653 suspended  sleep 1000

$ nohup sleep 2000 &
[1] 18745
nohup: appending output to 'nohup.out'

$ jobs
[1]-  Suspended              sleep 10000
[2]+  Running                nohup sleep 2000 &

$ bg %1 # 将作业 %1 (sleep 10000) 转到后台继续执行
[1]+ sleep 10000 &

$ jobs
[1]+  Running                sleep 10000 &
[2]-  Running                nohup sleep 2000 &

$ kill -STOP %1 # 发送 SIGSTOP 信号给作业 %1
[1]+  Stopped (signal)       sleep 10000

$ jobs
[1]+  Stopped (signal)       sleep 10000
[2]-  Running                nohup sleep 2000 &

$ kill -SIGHUP %1 # 发送 SIGHUP 给作业 %1 (通常导致程序终止)
[1]+  Hangup                 sleep 10000

$ jobs # 作业1已终止
[2]+  Running                nohup sleep 2000 &

$ kill -SIGHUP %2 # nohup 包装的进程会忽略 SIGHUP
# (通常没有输出，进程继续运行)

$ jobs
[2]+  Running                nohup sleep 2000 &

$ kill %2 # 默认发送 SIGTERM
[2]+  Terminated             nohup sleep 2000

$ jobs # 所有作业都已结束

注意: jobs 的输出格式和作业状态指示符 (+, -) 可能因 Shell 和配置而异。示例中的 PID 和作业号仅为演示。

终端多路复用器 (Terminal Multiplexers)

在命令行工作时，我们经常希望同时运行和查看多个任务。虽然可以打开多个终端窗口，但使用终端多路复用器（如 tmux）是更通用和强大的解决方案。

tmux 允许你在单个终端窗口内创建和管理多个 Shell 会话，这些会话可以组织在窗格 (panes) 和窗口 (windows) 中。更重要的是，你可以从一个 tmux 会话中分离 (detach)*，然后在稍后重新连接 (attach)，即使关闭了原来的终端窗口，会话及其中的进程也会继续运行。这对于远程服务器工作尤其有用。

tmux 基础

tmux 的操作主要通过快捷键完成，默认的前缀键是 Ctrl-b (按下 Ctrl+b，释放，然后再按后续键)。

会话 (Sessions)

  • tmux 或 tmux new: 启动新会话。
  • tmux new -s <name>: 命名新会话。
  • tmux ls: 列出所有会话。
  • <Ctrl-b> d: 分离当前会话。
  • tmux a: 重新连接最后分离的会话。
  • tmux a -t <name>: 连接指定会话。

窗口 (Windows)

  • <Ctrl-b> c: 创建新窗口。
  • <Ctrl-b> <0-9>: 跳转到第 N 个窗口。
  • <Ctrl-b> p: 上一个窗口。
  • <Ctrl-b> n: 下一个窗口。
  • <Ctrl-b> ,: 重命名当前窗口。
  • <Ctrl-b> w: 列出所有窗口 (可选择)。
  • <Ctrl-b> &: 关闭当前窗口 (需确认)。

窗格 (Panes)

  • <Ctrl-b> %: 垂直分割。
  • <Ctrl-b> ": 水平分割。
  • <Ctrl-b> <方向键>: 移动到指定方向窗格。
  • <Ctrl-b> o: 循环切换窗格。
  • <Ctrl-b> z: 缩放/恢复当前窗格。
  • <Ctrl-b> [: 进入复制模式 (向上滚动/搜索)。
  • <Ctrl-b> ]: 粘贴复制内容。
  • <Ctrl-b> x: 关闭当前窗格 (需确认)。

复制模式 (Copy Mode)

进入复制模式 (<Ctrl-b> [ 或 <Ctrl-b> PageUp) 后：

  • 使用类似 Vi 的键位移动: hjkl, Ctrl+u/d, gg, G。
  • / 进行搜索。
  • 按 Space 开始选择，移动光标扩展选区。
  • 按 Enter 复制选区到 tmux 粘贴板并退出复制模式。
  • q 退出复制模式。

更多资源: tmux 快速教程, 更详细的文章 (也包含 screen 命令)。 screen 是另一个较早的终端多路复用器，其前缀键是 Ctrl-a。

别名 (Aliases)

键入包含许多选项或冗长参数的长命令可能会很繁琐。大多数 Shell 都支持别名 (aliasing) 功能，允许你为长命令创建简短的替代品。

Shell 别名是另一个命令的简写，当你在 Shell 中输入别名时，它会自动被替换为完整的命令。例如，在 bash 中定义别名的基本语法是：

alias alias_name="command_to_alias arg1 arg2"

注意：等号 = 周围没有空格。

别名的妙用

示例:

  • 为常用选项创建简写: 
    alias ls='ls --color=auto' # 大多数发行版默认配置
alias ll='ls -alFh'      # F:分类, a:所有, l:长格式, h:人类可读
alias l.='ls -d .* --color=auto' # 仅列出隐藏文件
  • 为常用命令节省输入: 
    alias gs='git status'
alias ga='git add'
alias gc='git commit'
alias v='vim' # 或 nvim
  • 防止输错命令 (幽默示例): 
    alias sl='ls' # 如果你不小心输入 sl
  • 覆盖现有命令以获得更好/更安全的默认行为: 
    alias mv='mv -i'         # -i: 覆盖前提示
alias cp='cp -i'
alias rm='rm -I'         # -I: 删除多个文件或递归删除时提示一次 (更安全)
alias mkdir='mkdir -p'   # -p: 按需创建父目录
alias df='df -h'         # -h: 人类可读格式
alias du='du -h'

管理别名

  • 要临时忽略一个别名，在其前面加上反斜杠 \ (例如 \ls)。
  • 使用 unalias alias_name 可以完全禁用一个别名。
  • 输入 alias alias_name 可以查看该别名的定义。
  • 只输入 alias 会列出所有当前已定义的别名。

注意：默认情况下，别名仅在当前 Shell 会话中有效。要使别名持久化，需要将它们添加到 Shell 的启动配置文件中（例如 .bashrc 或 .zshrc），我们将在下一节讨论 Dotfiles 时详细介绍。

Dotfiles (配置文件)

许多程序使用纯文本文件进行配置，这些文件通常被称为 dotfiles，因为它们的文件名以点 . 开头 (例如 ~/.vimrc, ~/.bash_profile)，这使得它们在默认情况下被 ls 等命令隐藏。

Shell 就是一个典型的例子。启动时，你的 Shell 会读取一系列配置文件来加载其设置。这个过程可能根据 Shell 类型（如 bash, zsh）、是否为登录 Shell 或交互式 Shell 而有所不同。对于 bash，编辑 ~/.bashrc (用于交互式非登录 Shell) 或 ~/.bash_profile (用于登录 Shell，通常会加载 .bashrc) 是常见的做法。你可以在这些文件中包含启动时需要运行的命令，比如我们之前讨论的别名，或者修改 PATH 环境变量等。

常见使用 Dotfiles 的工具

  • bash: ~/.bashrc, ~/.bash_profile
  • zsh: ~/.zshrc, ~/.zprofile, ~/.zshenv
  • git: ~/.gitconfig, project/.git/config
  • vim: ~/.vimrc, ~/.vim/
  • nvim: ~/.config/nvim/init.vim (或 .lua)
  • ssh: ~/.ssh/config
  • tmux: ~/.tmux.conf

管理你的 Dotfiles

将你的 dotfiles 进行版本控制并集中管理有许多好处：

  • 易于安装： 在新机器上应用你的个性化设置非常快捷。
  • 可移植性： 确保你的工具在任何地方都以相同的方式工作。
  • 同步： 方便地在多台设备间同步和更新配置。
  • 变更跟踪： 对配置的修改进行版本控制，便于回顾和管理。

那么，应该在 dotfiles 中包含哪些内容呢？你可以通过阅读官方文档、man 页面或搜索相关的博客文章来了解特定工具的配置选项。查看其他人的 dotfiles 也是一个很好的学习途径 (例如 GitHub 上的 Dotfiles.GitHub.io 集合)。

本课程讲师的 dotfiles 示例: Anish, Jon, Jose.

组织和部署 Dotfiles 的方法

推荐将所有 dotfiles 存放在一个专用文件夹中，进行版本控制 (如 Git)，然后使用脚本或工具将它们符号链接 (symlink) 到各自在主目录中的正确位置。

常用管理工具/方法:
  • 手动创建符号链接: 最直接，但文件多时不易管理。
  • 自定义脚本: 编写 Shell 脚本来自动化符号链接的创建。
  • 专用工具:
    • GNU Stow: 流行选择，按程序组织 dotfiles 子目录。
    • rcm: 另一个 dotfile 管理器。
    • chezmoi: 功能更强大，支持模板和密码管理。
    • 其他: yadm, dotbot 等。

Dotfiles 的可移植性

在多台具有不同操作系统或 Shell 的机器上工作时，确保 dotfiles 的可移植性是一个常见的挑战。某些配置可能在一台机器上有效，但在另一台机器上无效，或者你可能希望某些设置仅应用于特定机器。

处理可移植性的技巧

  • 条件逻辑: 在配置文件中使用条件语句 (如 Shell中的 if) 来应用特定于机器或操作系统的设置。 
    # 示例：根据操作系统设置不同的 ls 别名
if [[ "$(uname)" == "Linux" ]]; then
  alias ls='ls --color=auto -F'
elif [[ "$(uname)" == "Darwin" ]]; then # macOS
  alias ls='ls -GF'
fi

# 示例：根据主机名应用特定设置
if [[ "$(hostname)" == "myAwesomeServer" ]]; then
  export HTTP_PROXY="http://proxy.example.com:8080"
fi

# 示例：检查 Shell 类型 (zsh 特定设置)
# if [[ "$SHELL" == *"/zsh"* ]]; then 
#   source ~/.zsh_plugins/zsh-syntax-highlighting.zsh
# fi
  • 使用 Include 指令: 如果配置文件支持 (例如 ~/.gitconfig)，使用 include 指令加载特定于机器的配置片段。 
    # ~/.gitconfig
[include]
    path = ~/.gitconfig_local # 本地特定配置

    然后 ~/.gitconfig_local 文件可以包含特定于当前机器的设置，并且可以被主 dotfiles 仓库忽略或单独管理。

  • 共享配置片段: 对于希望在不同程序间共享的配置 (例如 bash 和 zsh 共享别名)，可以将它们写入一个单独的文件 (如 ~/.aliases)，然后在各自的配置文件中 source 这个文件。 
    # 在 .bashrc 或 .zshrc 中
if [ -f ~/.aliases ]; then
    source ~/.aliases
fi

远程计算机 (Remote Machines)

程序员在日常工作中经常需要与远程服务器进行交互，无论是部署应用还是利用更强大的计算资源。安全 Shell (SSH)是为此目的设计的关键工具，它允许你安全地连接和操作远程计算机。

SSH 基本连接

要通过 SSH 连接到服务器，基本命令格式如下：

ssh username@hostname_or_ip

例如，ssh foo@bar.mit.edu 会尝试以用户 foo 的身份连接到服务器 bar.mit.edu。服务器地址也可以是 IP 地址，如 ssh foobar@192.168.1.42。

远程执行命令

ssh 一个常被忽视的功能是直接在远程服务器上执行命令并将结果返回到本地终端，而无需完全登录远程 Shell：

# 在远程服务器的 foobar 用户主目录下执行 ls
ssh foobar@server ls

# 远程执行 ls，并在本地 grep 其输出
ssh foobar@server ls | grep PATTERN

# 将本地文件的内容通过管道传给远程服务器上的 grep 命令
cat localfile | ssh foobar@server "grep PATTERN"

理解数据流向 (本地 vs 远程执行) 对于构建有效的命令管道非常重要。

SSH 密钥认证

基于密钥的身份验证使用公钥密码学，允许你在不直接暴露私钥的情况下向服务器证明你拥有该私钥，从而避免每次连接都输入密码。你的私钥 (通常是 ~/.ssh/id_rsa 或更现代的 ~/.ssh/id_ed25519) 非常重要，务必妥善保管。

密钥生成

你可以使用 ssh-keygen 命令生成 SSH 密钥对：

ssh-keygen -o -a 100 -t ed25519 -f ~/.ssh/id_ed25519 -C "your_email@example.com"
# -o: 使用新的 OpenSSH 格式保存私钥 (更抗暴力破解)
# -a 100: KDF (密钥派生函数) 轮数，增加破解难度
# -t ed25519: 指定密钥类型为 Ed25519 (现代、安全、快速)
# -f ~/.ssh/id_ed25519: 指定密钥文件路径
# -C "your_email@example.com": 注释，通常是你的邮箱，方便识别

在生成过程中，强烈建议为你的私钥设置一个密码短语 (passphrase)*。这层额外的保护可以防止在你的私钥文件被他人获取时，对方能立即使用它。配合使用 ssh-agent 或 gpg-agent，你可以在会话开始时输入一次密码短语，之后无需重复输入。

要检查现有私钥是否设置了密码短语，可以运行: ssh-keygen -y -f /path/to/private_key_file。

在服务器上设置公钥

SSH 服务器通过检查目标用户主目录下的 ~/.ssh/authorized_keys 文件来确定允许哪些客户端公钥连接。要将你的公钥添加到远程服务器的此文件中，可以使用以下方法：

方法1: 手动复制 (确保权限正确)
cat ~/.ssh/id_ed25519.pub | ssh username@remote_server \
  'mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys'
方法2: 使用 ssh-copy-id (推荐)
ssh-copy-id -i ~/.ssh/id_ed25519.pub username@remote_server
# 如果你只有一个主要的公钥，可以省略 -i ~/.ssh/id_ed25519.pub

ssh-copy-id 会自动处理远程目录和文件的创建及权限设置。

通过 SSH 复制文件

有多种方法可以通过 SSH 在本地和远程计算机之间复制文件：

常用复制方法:

  • 1. ssh 配合 tee 或 cat (简单重定向):

    适用于将本地文件内容写入远程单个文件。

    # 将 localfile 的内容写入远程服务器上的 serverfile (tee 会同时输出到STDOUT)
cat localfile | ssh username@remote_server "tee serverfile"

# 或者，如果只想写入文件不看输出:
cat localfile | ssh username@remote_server "cat > serverfile"
  • 2. scp (Secure Copy):

    当需要复制大量文件或整个目录时，scp 更方便，支持递归复制。

    # 复制本地文件到远程
scp /path/to/local_file username@remote_host:/path/to/remote_file

# 从远程复制文件到本地
scp username@remote_host:/path/to/remote_file /path/to/local_file

# 递归复制目录 (使用 -r 选项)
scp -r local_directory/ username@remote_host:/remote_parent_directory/
  • 3. rsync (Remote Sync):

    rsync 是一个更强大和灵活的文件同步工具。它能通过比较本地和远程文件，仅传输差异部分，从而避免不必要的复制。它还支持从中断的传输中恢复，并提供对符号链接、权限等的细粒度控制。

    # 常用 rsync 命令示例 (将本地同步到远程)
rsync -avz --progress local_source_directory/ username@remote_host:remote_destination_directory/

# 常用选项解释:
#   -a: 归档模式 (递归，保留符号链接、权限、时间戳、所有者、组等)
#   -v: 详细输出模式
#   -z: 传输时压缩文件数据
#   --progress: 显示传输进度
#   --delete: 删除目标目录中源目录不存在的文件 (用于使两边完全一致)
# 注意源路径末尾的 / 会影响行为 (是否复制目录本身或其内容)

SSH 端口转发 (Port Forwarding)

在很多情况下，远程服务器上的软件会监听特定的网络端口，但这些端口可能无法从外部网络直接访问。SSH 端口转发允许你在本地计算机和远程服务器之间建立一个安全的通道，使得本地端口可以映射到远程服务器上的端口（反之亦然）。

本地端口转发 (Local Port Forwarding)

这是最常见的端口转发类型。它允许你将本地机器上的一个端口转发到 SSH 服务器可以访问的某个目标主机和端口（通常是 SSH 服务器自身上的服务）。

命令格式:

ssh -L local_port:destination_host:destination_port यूजर@ssh_server

（destination_host 通常是 localhost，指相对于 ssh_server 而言）

示例: 假设远程服务器 remote_server 上运行了一个 Jupyter Notebook，监听其自身的 localhost:8888。要通过本地机器的 localhost:9999 访问它，可以执行：

ssh -L 9999:localhost:8888 foobar@remote_server

之后，在本地浏览器中访问 http://localhost:9999 即可连接到远程的 Jupyter Notebook。

远程端口转发 (Remote Port Forwarding)

这种方式不太常见，但用于将远程 SSH 服务器上的某个端口转发到你本地机器（或本地网络中其他机器）上的服务。这允许外部通过该 SSH 服务器访问你本地的服务。

命令格式:

ssh -R remote_server_port:destination_host:destination_port यूजर@ssh_server

（destination_host 通常是 localhost，指相对于你执行 ssh 命令的本地机器而言）

示例: 假设你的本地开发机器上有一个 Web 服务运行在 localhost:3000。你想让远程服务器 ssh_server 上的 8080 端口能够访问到这个本地服务：

ssh -R 8080:localhost:3000 user@ssh_server

现在，在 ssh_server 上访问 localhost:8080 的请求会被转发到你本地机器的 localhost:3000。

注意：为了使远程端口能被 ssh_server 以外的机器访问，可能需要在服务器的 /etc/ssh/sshd_config 文件中将 GatewayPorts 设置为 yes 或 clientspecified。

SSH 配置文件 (~/.ssh/config)

对于复杂的 SSH 连接参数 (如指定端口、密钥文件、端口转发等)，反复在命令行输入会非常繁琐。虽然可以使用 Shell 别名，但 SSH 自身提供了一个更强大和集中的解决方案：配置文件 ~/.ssh/config。

通过在这个文件中定义主机及其参数，你可以简化 SSH 命令，例如用 ssh my_server 代替一长串复杂的命令。更重要的是，其他依赖 SSH 的工具（如 scp, rsync, git）也能读取和使用这些配置。

~/.ssh/config 示例与常用选项

# ~/.ssh/config

Host vm_alias  # 定义一个主机别名
    User foobar
    HostName 172.16.174.141  # 实际的 IP 地址或域名
    Port 2222                # 自定义 SSH 端口
    IdentityFile ~/.ssh/id_ed25519_vm # 为此主机指定特定密钥文件
    LocalForward 9999 localhost:8888   # 配置本地端口转发
    # RequestTTY yes                   # 有时需要强制分配 TTY (伪终端)
    # ServerAliveInterval 60           # 每60秒发送心跳包保持连接

# 可以使用通配符配置一组主机
Host *.mit.edu
    User your_mit_username
    # ForwardAgent yes                 # 谨慎使用：允许远程主机使用本地 ssh-agent 验证其他主机
    # ControlMaster auto               # 启用 SSH 连接共享
    # ControlPath ~/.ssh/cm-%r@%h:%p   # 连接共享的控制路径
    # ControlPersist 600               # 保持主连接在后台600秒，加快后续连接

Host github.com
    User git
    HostName github.com
    IdentityFile ~/.ssh/id_github    # GitHub 使用的特定密钥
    # IdentitiesOnly yes             # 仅使用 IdentityFile 指定的密钥进行认证
关键点:
  • Host: 定义一个配置块的开始，后面跟一个主机别名或模式。
  • HostName: 指定实际连接的主机名或 IP 地址。
  • User: 指定连接时使用的用户名。
  • Port: 指定远程 SSH 服务器的端口（默认为 22）。
  • IdentityFile: 指定用于身份验证的私钥文件路径。
  • LocalForward / RemoteForward: 配置端口转发。
  • ForwardAgent: 是否允许转发 ssh-agent 的连接。
  • ControlMaster, ControlPath, ControlPersist: 用于 SSH 连接复用，可以显著加快对同一主机的后续连接速度，因为它复用已建立的 TCP 连接。

当你将 ~/.ssh/config 文件作为 dotfiles 的一部分并公开分享时，请注意其中可能包含的敏感信息，如服务器地址、用户名、开放端口等，这些信息可能被恶意利用。

服务器端的 SSH 配置通常在 /etc/ssh/sshd_config 文件中进行。在这里可以进行更全局的设置，如禁用密码认证、更改 SSH 监听端口、启用或禁用 X11 转发等。

SSH 相关工具与技巧

Mosh (Mobile Shell)

当网络连接不稳定或需要在不同网络间切换（例如笔记本电脑合盖再打开）时，标准的 SSH 连接很容易断开，这会非常令人沮丧。Mosh 是 SSH 的一个优秀替代品和补充，它提供了更强大的连接鲁棒性：

  • 支持漫游连接和间歇性连接：即使 IP 地址改变或网络暂时中断，Mosh 也能尝试保持或恢复会话。
  • 智能本地回显：对于输入字符等操作，Mosh 会在本地立即显示，减少了高延迟连接下的卡顿感。
  • 通常与 SSH 协同工作进行初始认证，然后通过 UDP 建立自己的连接。

sshfs (SSH Filesystem)

sshfs 是一个基于 FUSE (Filesystem in Userspace) 的工具，它允许你将远程服务器上的一个目录挂载到本地文件系统的一个挂载点上。一旦挂载成功，你就可以像操作本地文件和目录一样，使用本地的编辑器、文件管理器等工具来访问和修改远程文件。

# 示例：将远程服务器的 ~/remote_projects 目录挂载到本地的 ~/mnt/server_projects
mkdir -p ~/mnt/server_projects
sshfs username@remote_server:~/remote_projects ~/mnt/server_projects

# 完成后卸载:
fusermount -u ~/mnt/server_projects

Shells 和框架 (Shells & Frameworks)

虽然我们主要介绍了 bash Shell，因为它是最普遍且通常是系统默认的 Shell，但存在许多其他的 Shell 可供选择，它们提供了不同的功能和用户体验。

Zsh (Z Shell)

Zsh 是 bash 的一个流行替代品，它基本上是 bash 的超集，并内置了许多便捷功能，例如：

  • 更智能的通配符展开 (Globbing)，如 **/* 可递归匹配。
  • 内联通配符展开。
  • 拼写纠正建议。
  • 改进的 Tab 自动补全和选择界面。
  • 路径展开 (例如输入 cd /u/lo/b 然后按 Tab 可能展开为 /usr/local/bin)。
  • 更强大的数组和哈希表处理能力。

Shell 框架 (Frameworks)

框架可以进一步增强你的 Shell 功能和外观。一些流行的通用框架（主要针对 Zsh）包括:

  • Prezto
  • Oh My Zsh

还有许多专注于特定功能的小型插件/框架，例如：

  • zsh-syntax-highlighting (命令语法高亮)
  • zsh-autosuggestions (基于历史记录的灰色自动建议)
  • zsh-history-substring-search (输入部分命令按上下键搜索历史)

Fish Shell (Friendly Interactive Shell)

Fish 是另一个以用户友好为设计目标的现代 Shell。它开箱即用地包含了许多 Zsh 需要通过插件实现的功能，例如：

  • 默认启用的语法高亮。
  • 智能且即时的自动建议 (灰色显示)。
  • 基于 Man 手册的参数自动补全。
  • 更简洁的脚本语法 (可能与 POSIX Shell 不完全兼容)。
  • 丰富的提示符主题和配置选项。

在使用功能丰富的 Shell 或添加大量框架/插件时，需要注意它们可能会增加 Shell 的启动时间和响应延迟。如果感觉到卡顿，可以尝试分析配置 (time zsh -i -c exit 可以粗略测量启动时间) 并禁用不常用或影响性能的功能。

终端模拟器 (Terminal Emulators)

除了自定义 Shell 本身，花时间选择并配置你的终端模拟器 (Terminal Emulator) 也同样重要。终端模拟器是你与 Shell 交互的图形界面窗口。

市面上有许多不同的终端模拟器可供选择 ([这里有一个比较列表](https://wiki.archlinux.org/title/List_of_terminal_emulators))。考虑到你可能会在终端上花费大量时间，研究并调整其设置以符合你的偏好和工作流程是非常值得的。

值得考虑的终端配置方面

  • 字体选择: 使用适合编程的等宽字体，例如支持连字 (ligatures) 的 Fira Code, JetBrains Mono, Cascadia Code。
  • 配色方案: 选择一个你喜欢且对比度合适的配色主题，如 Solarized, Dracula, Gruvbox, Nord 等。
  • 键盘快捷键: 自定义复制、粘贴、新建标签页/窗口等操作的快捷键。
  • 标签页/窗格支持: 虽然 tmux 功能更强，但一些终端也内置了基本的标签页和窗格管理。
  • 回滚配置: 调整终端保留的历史输出行数 (scrollback buffer size)。
  • 性能: 一些较新的终端模拟器利用 GPU 加速来提高渲染速度和响应性。

一些现代终端模拟器示例:

  • Alacritty (跨平台, GPU 加速, 配置驱动)
  • kitty (跨平台, GPU 加速, 功能丰富, 可扩展)
  • WezTerm (跨平台, GPU 加速, Lua 配置, 内置 Multiplexing)
  • Hyper (基于 Web 技术构建, 跨平台, 可定制性强)
  • Windows Terminal (现代 Windows 终端，支持多 Shell, GPU 加速)

练习 (Exercises)

  1. 1. 作业控制:

    • 启动一个 sleep 10000 作业，用 Ctrl+Z 将其置于后台，并用 bg 继续执行。然后使用 pgrep 找到其 PID，并使用 pkill 在不直接输入 PID 的情况下终止它 (提示: pgrep -af 和 pkill -af)。
    • 如何让一个进程等待另一个后台进程完成后再执行？例如，启动 sleep 60 &，然后让 ls 等待它完成后执行 (提示: wait 命令和 $! 变量)。
    • 编写一个名为 pidwait 的 bash 函数，它接受一个 PID 作为参数，并阻塞直到该 PID 对应的进程完成。使用 kill -0 <PID> (检查进程是否存在，成功退出码为 0) 和 sleep 来避免 CPU 浪费。
    查看解答提示 
    # 终止 sleep 进程
sleep 10000
# Ctrl-Z
bg
pgrep -af "sleep 10000" # 找到 PID
pkill -af "sleep 10000" # 终止

# 等待后台进程
sleep 60 &
wait $! # $! 是最后一个后台作业的 PID
ls

# pidwait 函数
pidwait() {
    local pid_to_wait_for="$1"
    if [ -z "$pid_to_wait_for" ]; then echo "Usage: pidwait "; return 1; fi
    echo "Waiting for PID $pid_to_wait_for to complete..."
    while kill -0 "$pid_to_wait_for" 2>/dev/null; do
        sleep 1
    done
    echo "PID $pid_to_wait_for has completed."
}
# 测试: sleep 5 &; pidwait $!; echo "Done"

  3. 3. 别名:

    • 创建一个别名 dc，使其在你错误输入时等同于 cd。
    • 找出你最常用的 10 个命令，并为它们设置更短的别名。
      (提示: Bash 用 history | sed 's/^[ ]*[0-9]*[ ]*//' | sort | uniq -c | sort -nr | tail -10； Zsh 用 history -E 1 0 | sed 's/^[ ]*[0-9]*[ ]*//' | sort | uniq -c | sort -nr | tail -10 或类似方法分析历史记录)
    查看解答提示 
    # dc 别名
alias dc='cd'

# 查找常用命令 (Bash 示例)
history | sed 's/^[ ]*[0-9]*[ ]*//' | sort | uniq -c | sort -nr | tail -n 10
# 然后根据输出创建别名，例如：
# alias gco='git checkout'
# alias ..='cd ..'

  4. 4. Dotfiles:

    • 创建 ~/dotfiles 目录并用 Git 初始化。
    • 至少为一个程序（如 Shell, Git, Vim）添加配置文件，并进行一些个性化设置（例如自定义 Shell 提示符 $PS1）。
    • 设置一种在新机器上快速安装/部署 dotfiles 的方法（例如使用简单的符号链接脚本或 Stow, chezmoi 等工具）。
    • 在一个新的虚拟机或容器中测试你的安装脚本。
    • 将你当前使用的所有工具的配置迁移到你的 dotfiles 仓库中。
    • (可选) 在 GitHub 或类似平台上发布你的 dotfiles。

  5. 5. 远程计算机:

    • (准备) 安装一个 Linux 虚拟机 (如使用 VirtualBox)。
    • 检查或生成 SSH 密钥对 (ssh-keygen -t ed25519)，建议设置密码短语并使用 ssh-agent。
    • 配置 ~/.ssh/config，添加一个指向你虚拟机的条目，包含 Host, User, HostName, IdentityFile，并设置 LocalForward 9999 localhost:8888。
    • 使用 ssh-copy-id vm_alias 将公钥复制到虚拟机。
    • 在虚拟机中启动一个简单的 Web 服务器 (python3 -m http.server 8888)。
    • 在本地机器浏览器中访问 http://localhost:9999，验证端口转发是否成功。
    • (安全实践) 编辑虚拟机上的 /etc/ssh/sshd_config，设置 PasswordAuthentication no 和 PermitRootLogin no，然后重启 SSH 服务 (sudo systemctl restart sshd 或 sudo service ssh restart)。再次尝试 SSH 连接。
    • (挑战) 在 VM 和本地都安装 mosh，尝试建立 Mosh 连接。然后断开 VM 的网络，再恢复，观察 Mosh 是否能恢复连接。
    • (挑战) 查阅 ssh 的 -N 和 -f 标志，找出一个可以在后台运行端口转发的命令。
    查看挑战解答提示

    后台端口转发命令:

    ssh -N -f -L local_port:destination_host:destination_port user@ssh_server
# -N: 不执行远程命令。
# -f: 请求 ssh 在命令执行前进入后台。
